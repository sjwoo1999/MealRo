# MealRo Figma 204:694 비교 분석

## 문서 목적

이 문서는 Figma `MealRo / node 204:694` 와이어프레임과 현재 구현된 MVP를 비교해서,

- 현재 어디까지 맞아 있는지
- 어떤 흐름이 빠져 있는지
- 다음 구현 우선순위가 무엇인지

를 빠르게 판단하기 위한 문서다.

기준 자료:

- Figma `MealRo / node 204:694`
- 현재 코드베이스 `src/app`, `src/components`

## 한 줄 결론

현재 구현은 `카메라 기반 기록 -> 분석 -> 저장 -> 추천/탐색 -> 피드/상세`까지는 닿아 있지만,
Figma가 보여주는 핵심 루프인 `추천 통합`, `식당 상세/도착 확인`, `분석 리포트`는 아직 비어 있다.

## 현재 정렬 상태

### 이미 맞아 있는 축

- 하단 카메라 FAB 기반 스캔 진입
- 식사 시간 선택
- 음식 사진 업로드와 AI 분석
- 분석 결과 확인과 수정
- 기록 저장
- 추천 화면 진입
- 주변 탐색 화면 진입
- 기록 피드와 기록 상세 보기

### 덜 맞아 있는 축

- 추천과 탐색이 한 화면 흐름으로 이어지지 않음
- 식당 선택 이후의 상세 의사결정이 약함
- 도착 확인 루프가 없음
- 하단 `분석` 영역이 실제 리포트로 열리지 않음

## 피그마 기준 핵심 흐름

Figma `204:694`에서 읽히는 중심 플로우는 아래와 같다.

1. 카메라 FAB 탭
2. 식사 시간 선택
3. 촬영
4. AI 분석 결과 확인
5. 다음 행동 분기
   - 역추산 추천
   - 위치 기반 메뉴/식당 탐색
6. 지도에서 후보 확인
7. 식당 상세 확인
8. 도착 여부 확인
9. 분석/리포트로 회고

즉 피그마는 `기록 앱`이 아니라 `기록 -> 다음 식사 결정 -> 실제 방문 -> 회고`까지 이어지는 제품을 그리고 있다.

## 주요 차이점

### 1. 추천과 탐색이 아직 분리돼 있다

피그마는 `역추산 추천`, `위치 기반 추천`, `가격/거리/영업시간`이 하나의 의사결정 묶음처럼 보인다.
현재는 `/meal` 과 `/nearby` 가 분리돼 있어서 사용자가 추천을 보고 다시 탐색으로 건너가야 한다.

영향:

- 추천이 실제 식당 선택으로 바로 이어지지 않음
- 위치/영업시간/가격 요소가 추천 안에서 같이 읽히지 않음

### 2. Nearby는 탐색까지만 있고 결정 루프가 약하다

현재 `/nearby` 는 지도, 리스트, 선택 식당까지는 있다.
하지만 피그마처럼 `식당 상세 -> 메뉴 확인 -> 도착 체크`로 닫히지는 않는다.

영향:

- 사용자가 실제로 어디를 갈지 결정하는 경험이 약함
- `분석 -> 실사용 행동` 전환이 중간에서 끊김

### 3. 회고 화면이 비어 있다

피그마 하단 `분석` 영역은 단순 피드가 아니라 회고/리포트 성격이다.
현재는 베타 운영 편의상 공용 피드가 열려 있고, 인사이트 화면은 아직 없다.

영향:

- 서비스가 `기록/구경` 중심처럼 보일 수 있음
- 장기 가치인 `건강 관리` 축이 약하게 전달됨

### 4. FAB 중심 진입은 살아 있지만 메인 허브 구조는 다르다

피그마는 카메라 FAB가 매우 강한 중심이다.
현재는 홈에서 끼니를 고르는 허브가 먼저 보이고, FAB는 보조 진입으로 남아 있다.

영향:

- 첫 경험이 `즉시 촬영`보다 `시작 페이지 탐색` 쪽으로 살짝 이동함

### 5. 피드는 피그마 의도와 다르게 베타 운영용으로 변형됐다

현재 `/history`, `/feed` 는 베타 테스터 공용 기록 피드로 잘 작동한다.
하지만 이것은 피그마의 `개인 분석/회고` 흐름과는 별도 목적이다.

영향:

- 테스트 운영에는 유리함
- 제품 원래 흐름과는 다소 다름

## 심각도 기준 우선순위

### P0. 먼저 채워야 하는 것

1. `Nearby 상세 결정 화면`
식당 카드 선택 다음에 메뉴/거리/영업시간/예상 칼로리/CTA를 보여주는 상세 화면이 필요하다.

2. `도착 확인 플로우`
피그마에 있는 `도착했나요?` 단계가 빠져 있다.
이 단계가 들어가야 탐색이 실제 행동으로 닫힌다.

3. `추천과 탐색 연결 강화`
`/meal` 에서 추천안 선택 후 `/nearby` 로 가는 수준을 넘어서,
추천 결과 안에서 위치/가격/영업시간 정보를 일부 같이 보여주는 구조가 필요하다.

### P1. 다음 단계

1. `인사이트 페이지 복구`
최소한 주간 요약, 최근 기록, 칼로리/탄단지 누적 정도는 다시 열어야 한다.

2. `홈/FAB 진입 구조 재정렬`
홈 허브를 줄이고 FAB 기반 즉시 기록성을 더 살릴지 판단이 필요하다.

3. `피드 역할 재정의`
`베타 피드`를 유지할지, `개인 리포트`와 분리할지 정해야 한다.

### P2. 나중 단계

1. 프랜차이즈/배달/가격 민감도 추천 강화
2. 커뮤니티/공개 피드 고도화
3. 시각 리포트와 차트 고도화

## 화면별 판정

| 화면 | 피그마 적합도 | 코멘트 |
| --- | --- | --- |
| `/` | 중간 | 식사 선택은 맞지만, FAB 중심 강도는 피그마보다 약함 |
| `/scan` | 높음 | 식사 선택 후 촬영/분석 진입 구조는 비교적 잘 맞음 |
| 분석 결과 | 높음 | 수정/저장/다음 행동 분기가 잘 열려 있음 |
| `/meal` | 중간 | 역추산 추천은 맞지만 탐색 요소 통합이 약함 |
| `/nearby` | 중간 이하 | 지도/리스트는 있으나 상세/도착 루프가 없음 |
| `/history`, `/feed` | 중간 | 운영용 피드로는 좋지만 피그마 회고 흐름과는 다름 |
| `/insights` | 낮음 | 아직 미구현 |

## 바로 실행할 구현 순서

1. `/nearby/[id]` 또는 동등한 식당 상세 화면 추가
2. `/nearby` 에서 `도착 확인` 화면 또는 상태 추가
3. `/meal` 결과에 `위치/영업시간/가격` 요약 카드 추가
4. `/insights` 최소 버전 복구
5. 홈에서 `FAB 우선 진입`을 더 강하게 만들지 결정

## 결론

현재 MVP는 피그마 대비 `앞단 플로우`는 꽤 살아 있다.
하지만 Figma의 차별점은 단순 기록이 아니라 `추천과 탐색을 통해 실제 식사 결정을 돕고, 이후 회고까지 이어지는 것`이다.

따라서 다음 구현 기준은 단순하다.

- `scan`을 더 다듬는 것보다
- `meal -> nearby -> detail -> arrived -> insights`

이 후반 루프를 먼저 닫는 쪽이 피그마와 더 가까워진다.
