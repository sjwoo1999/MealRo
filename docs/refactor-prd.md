# MealRo Refactor PRD

## 1. 문서 개요

이 문서는 MealRo의 현재 리팩토링과 MVP 정렬 작업을 위한 실행 문서다.

기존 PRD가 제품 아이디어와 전체 범위를 폭넓게 설명한다면, 이 문서는 다음에 집중한다.

- 현재 Figma 보드 기준 MVP 핵심이 무엇인지
- 현재 코드베이스가 어디까지 와 있는지
- 어떤 순서로 리팩토링해야 실제 구현 속도가 빨라지는지

기준 참고 자료:

- `docs/mvp-feature-priority.md`
- `docs/mvp-gap-analysis.md`
- `docs/refactor-roadmap.md`
- `docs/design-system-foundation.md`
- Figma `MealRo / node 204:694`

## 2. 배경

현재 프로젝트는 다음 특징을 동시에 가진다.

- 제품 기획은 비교적 많이 정리되어 있다.
- 실제 구현도 적지 않게 진행되어 있다.
- 하지만 화면, 기능, 상태, 스타일이 제품 흐름 기준으로 아직 완전히 정리되지는 않았다.

특히 Figma 보드에서는 `분석 -> 추천 -> 지도/상세 -> 회고`라는 연속된 사용자 경험이 보이지만,
현재 코드베이스는 그 기능들이 여러 화면과 컴포넌트에 나뉘어 존재한다.

따라서 지금의 핵심 과제는 새로운 기능을 무작정 추가하는 것이 아니라, 이미 존재하는 기능 자산을 MVP 방향에 맞춰 다시 정렬하는 것이다.

## 3. 문제 정의

### 3.1 제품 구조 문제

- 사용자 여정이 화면 단위로 끊겨 있다.
- 분석 이후의 다음 행동이 강하게 이어지지 않는다.
- 위치 추천, 상세, 저장, 회고가 하나의 흐름으로 보이지 않는다.

### 3.2 UI 시스템 문제

- 스타일과 레이아웃 규칙이 페이지마다 다르다.
- 공용 컴포넌트가 있지만 사용 강제력이 약하다.
- 아직 디자인 시스템이 확정되지 않은 상태에서 화면별 개별 구현이 계속되면 비용이 커진다.

### 3.3 구현 상태 문제

- 일부 기능은 이미 구현됐지만 저장/복구/이력/추천 연결이 느슨하다.
- 일부 화면은 mock 또는 placeholder 수준이다.
- API와 상태 관리도 향후 정리가 필요하다.

## 4. 제품 목표

이번 리팩토링의 목표는 MealRo를 다음과 같은 MVP 경험으로 재정렬하는 것이다.

### 목표

- 사용자는 빠르게 첫 식사 분석 결과를 볼 수 있어야 한다.
- 분석 이후 바로 추천 또는 탐색으로 이동할 수 있어야 한다.
- 추천은 사용자 목적, 식사 맥락, 위치 같은 요소를 점진적으로 반영해야 한다.
- 게스트 사용자는 기록을 잃지 않고 회원 전환할 수 있어야 한다.
- 이후 화면 확장이 가능하도록 공통 셸과 디자인 토큰을 먼저 고정해야 한다.

### 비목표

- 지금 단계에서 모든 소셜/피드 기능을 완성하지 않는다.
- 정교한 차트 시스템을 먼저 만들지 않는다.
- Figma의 모든 시안을 픽셀 단위로 바로 옮기지 않는다.
- 추천 알고리즘의 모든 경우의 수를 한 번에 해결하지 않는다.

## 5. MVP 범위 정의

## P0. Capture

- 식사 시간 선택
- 카메라/FAB 진입
- 업로드/촬영
- AI 분석
- 결과 화면

## P1. Decision

- 역추산 추천
- 사용자 목적 기반 추천 반영
- 결과 저장과 다음 행동 CTA

## P1. Explore

- 위치 기반 추천
- 추천 리스트
- 식당/메뉴 상세

## P2. Retention

- 이력
- 주간/월간 분석 화면
- 계정 전환/복구

## 6. 현재 상태 평가

현재 코드 기준으로 보면:

- `분석`: 구현 자산이 강함
- `역추산 추천`: 구현 자산이 있음
- `인증/게스트 저장`: 구현 자산이 있음
- `지도/추천 탐색`: 부분 구현
- `피드`: 미구현에 가까움
- `분석 대시보드`: 목업 성격이 강함

즉, 프로젝트는 "처음부터 만드는 상태"가 아니라 `핵심 자산은 이미 있고, 연결 경험과 구조가 부족한 상태`다.

## 7. 리팩토링 원칙

### 원칙 1. 기능보다 흐름을 먼저 정렬한다.

분석, 추천, 지도, 이력을 각자 예쁘게 만드는 것보다, 사용자가 어디서 들어와서 어디로 이동하는지가 먼저 정리되어야 한다.

### 원칙 2. 페이지보다 시스템을 먼저 만든다.

디자인 시스템이 완성되지 않았기 때문에, 먼저 토큰과 셸, 공용 컴포넌트를 고정해야 한다.

### 원칙 3. 이미 있는 기능은 버리지 않고 재배치한다.

현재 코드에 있는 분석/추천/인증 자산을 최대한 활용한다.

### 원칙 4. MVP 이후 기능은 명확히 후순위로 둔다.

피드, 고급 분석, 세부 소셜 기능은 현재 핵심 플로우를 흔들지 않는 범위에서 다룬다.

## 10. 성공 기준

이번 PRD 기준 성공은 아래와 같이 정의한다.

### 제품 흐름

- 사용자가 홈 또는 FAB에서 분석 플로우로 자연스럽게 진입할 수 있다.
- 분석 완료 후 추천 또는 탐색으로 바로 이동할 수 있다.
- 추천과 지도/상세가 서로 끊기지 않는다.

### 구현 구조

- 공통 셸과 공용 컴포넌트 사용 비율이 높아진다.
- 페이지별 하드코딩 스타일이 줄어든다.
- mock과 실데이터 경계가 명확해진다.

### 유지보수성

- 새 Figma 화면이 와도 기존 규칙 안에서 빠르게 수용할 수 있다.
- 기능 추가가 화면 복붙이 아니라 패턴 확장으로 가능해진다.

## 11. 리스크

### 1. Figma 변경 가능성

전체 화면이 아직 확정되지 않았기 때문에, 페이지 단위 완성보다 시스템 단위 정리가 우선이어야 한다.

### 2. 기능 자산 중복

이미 구현된 기능을 무시하고 다시 만들면 비용이 증가한다.

### 3. 상태 관리 분산

Auth, 온보딩, 익명 저장, 추천 데이터가 분산되어 있어 향후 책임 정리가 필요하다.

### 4. MVP 범위 팽창

지도, 추천, 피드, 분석 리포트를 동시에 다 완성하려 하면 흐름이 깨질 수 있다.

## 12. 즉시 다음 작업

이 문서 기준으로 당장 이어서 할 일은 다음 순서가 적절하다.

1. `home + scan`을 MVP Capture 흐름 기준으로 재구성
2. `meal/planner`를 Decision 흐름 기준으로 재구성
3. `nearby + item detail`을 Explore 흐름 기준으로 재구성
4. `history + insights`를 Retention 흐름 기준으로 재구성

## 13. 결론

MealRo의 현재 리팩토링은 "디자인을 예쁘게 다시 칠하는 작업"이 아니라,
이미 존재하는 분석/추천/저장 자산을 Figma 기준 MVP 흐름에 맞춰 다시 조직하는 작업이다.

따라서 앞으로의 판단 기준은 단순하다.

- 이 기능이 `분석 -> 추천 -> 탐색 -> 회고` 흐름을 강화하는가
- 이 화면이 디자인 시스템과 공통 셸 위에서 확장 가능한가

이 두 질문에 답하는 방향으로 구현을 진행한다.
