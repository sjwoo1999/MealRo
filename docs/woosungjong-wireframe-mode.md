# 우성종 와이어프레임 기준 작업 원칙

## 목적

이 문서는 앞으로 `우성종 파트`를 작업할 때, 현재 구현 화면이 아니라 `우성종이 작성한 와이어프레임`을 우선 기준으로 삼기 위한 작업 원칙 문서다.

핵심은 다음 한 줄이다.

- `Capture 트랙은 기존 화면 재사용보다 우성종 와이어프레임 구조를 먼저 따른다.`

## Source of Truth

우성종 파트 작업에서 우선 기준은 아래 순서로 둔다.

1. Figma의 우성종 와이어프레임 보드
2. `docs/woosungjong-scope.md`
3. `docs/refactor-prd.md`
4. 현재 코드 구현 상태

즉, 코드에 이미 화면이 있어도 와이어프레임 구조와 다르면 코드가 아니라 와이어프레임을 기준으로 수정한다.

## 적용 범위

이 원칙은 아래 범위에만 우선 적용한다.

- `src/app/page.tsx`
- `src/app/scan/page.tsx`
- `src/components/FoodScanner.tsx`
- `src/components/scan/FoodAnalysisResult.tsx`
- `src/components/scan/FoodEditBottomSheet.tsx`
- `src/components/home/HomeOnboardingSection.tsx`
- `src/components/home/RestorePendingMeal.tsx`
- `src/components/layout/BottomNavigation.tsx`

즉, 우성종 파트는 `Home Entry -> Scan Entry -> Capture Flow -> Capture Recovery`에 한정한다.

## 구현 원칙

### 1. 기존 화면을 예쁘게 다듬는 방식으로 가지 않는다.

- 현재 화면 위에 카드나 색만 덧씌우는 방식은 피한다.
- 기존 화면이 있더라도 레이아웃 단위로 다시 짤 수 있어야 한다.

### 2. 와이어프레임은 구조를 우선한다.

- 섹션 이름
- 입력/출력 순서
- 버튼 위치
- 다음 행동 CTA

이 네 가지를 먼저 맞춘다.

### 3. 시각적으로도 와이어프레임처럼 보여야 한다.

- 장식용 gradient, marketing hero, 강조용 브랜드 표현은 최소화한다.
- 구조가 먼저 읽히도록 단순한 블록과 단계 표시를 우선한다.
- “기존 화면을 정리한 버전”처럼 보이면 실패다.

### 4. 우성종 파트에서는 Capture 언어를 고정한다.

반드시 아래 언어가 일관되게 보여야 한다.

- 지금 기록할 식사
- 식사 시간 선택
- 사진 업로드/촬영
- AI 분석 중
- 분석 결과 확인
- 기록 확정
- 다음 단계로 이동

### 5. 다음 트랙의 내용은 연결만 하고 확장하지 않는다.

우성종 파트에서는 아래를 완성하려 하지 않는다.

- 추천 상세
- 지도 상세
- 피드/커뮤니티
- 분석 대시보드 고도화

필요한 경우에는 링크나 CTA만 둔다.

## 화면별 기대 상태

### Home

- 사용자가 3초 안에 “지금 무엇을 해야 하는지” 이해해야 한다.
- 첫 CTA는 반드시 `기록/스캔`이어야 한다.
- 추천/탐색은 2차 CTA여야 한다.

### Scan

- 식사 시간 맥락이 먼저 보여야 한다.
- 업로드 전 기대값과 업로드 후 상태 변화가 명확해야 한다.
- 분석 결과와 다음 행동이 한 흐름으로 느껴져야 한다.

### Capture Recovery

- 복구 기능은 중요하지만 메인 흐름을 가리면 안 된다.
- 우선순위는 `기록 시작 > 복구 안내`다.

### Bottom Navigation

- 중앙 FAB는 `Capture` 중심 역할이어야 한다.
- 다른 탭보다 스캔 진입이 더 강하게 보여야 한다.

## 리뷰 기준

우성종 파트 작업 리뷰는 아래 질문으로 본다.

1. 이 화면이 우성종 와이어프레임 구조를 먼저 따르고 있는가
2. 기존 UI를 손본 수준이 아니라 실제로 다시 짠 화면인가
3. 사용자가 기록 시작점을 더 빨리 이해하는가
4. 다음 단계 CTA가 과하지 않고 정확하게 이어지는가

## 지금부터의 적용 방식

앞으로 우성종 파트 요청이 들어오면, 기본 가정은 아래와 같다.

- “현재 구현 유지”가 아니라 “우성종 와이어프레임 우선 적용”
- “디자인 개선”이 아니라 “와이어프레임 구조 정렬”
- “전체 앱”이 아니라 “Capture 트랙 우선”

## 결론

우성종 파트는 더 이상 기존 화면을 정리하는 작업으로 보면 안 된다.

앞으로는 아래처럼 판단한다.

- 현재 구현과 와이어프레임이 충돌하면 와이어프레임을 따른다.
- 구조가 흔들리면 시각 완성도보다 구조를 먼저 맞춘다.
- 우성종 파트가 끝난 뒤에야 Decision/Explore/Retention 확장을 다시 본다.
